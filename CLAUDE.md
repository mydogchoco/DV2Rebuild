# CLAUDE.md — 드래곤빌리지2 오프라인 재구현 프로젝트

> 이 파일은 Claude Code가 매 세션마다 읽는 프로젝트 컨텍스트입니다.
> 내용이 바뀌면 여기부터 갱신하세요.

## 1. 프로젝트 개요

- **목표**: 서비스 종료된 모바일 2D RPG "드래곤빌리지2"(하이브로, 2014)를,
  추출된 그래픽 에셋을 이용해 **싱글플레이 오프라인 게임으로 재구현**한다.
- **용도**: 개인 소장/플레이 전용. 배포·공유·수익화하지 않는다.
- **타깃 플랫폼**: **PC**. 원작은 APK(모바일)였으나 재구현판은 PC에서 작동·배포
  (**Windows exe** 등 PC 친화 형식). 입력은 마우스/키보드 우선(터치 가정 금지).
- **원작 엔진(추정)**: Cocos2d-x (plist 아틀라스 + Spine + Lua 기반으로 추정)
- **재구현 엔진**: Godot 4 (GDScript). Spine은 구버전 포맷이라 spine-godot 직접 사용
  불가 → **오프라인 변환기로 Godot 네이티브 씬 생성**(옵션 4b, `docs/spine_spike.md`).
- **상세 로드맵/역할분담/워크플로**: `docs/roadmap.md` (이 파일과 짝을 이룸)

### ⚠️ 프로젝트를 규정하는 핵심 사실
1. **에셋에는 게임플레이 데이터가 없다.** DV2/ 레포는 그래픽·오디오·파티클·셰이더·
   폰트·UI문자열뿐. 드래곤 스탯/스킬수치/스테이지/부화·조합/전투공식/성장곡선은
   서버에 있었고 사라졌다. → `data/*.json`은 **파싱이 아니라 역설계로 복원**한다.
2. **비주얼은 거의 다 있다.** 드래곤 baby/child/adult + 변형 spine, 스킬 `skill/{id}.png`
   등 키별로 존재. 설계 스켈레톤만 만들면 살을 붙일 그래픽은 대부분 확보돼 있다.
3. **오프라인 전용으로 변형**: 친구/PvP/경매/랭킹/서버이벤트/로그인 **삭제**,
   레이드 등 멀티 콘텐츠는 **솔로 플레이로 재설계**.

### 역할 분담
- **사용자**: 유실된 게임 시스템 지식 제공(스탯/공식/규칙, 웹서핑+자체밸런스), 밸런스·
  변형 방향 결정, 검수·플레이. 단 **비-게임개발자**(기본 프로그래밍 지식만).
- **Claude**: 그 지식을 설계문서·데이터스키마·코드로 구조화, 에셋 파이프라인/엔진 구현,
  게임 개발 순서·툴 안내. **"무엇을 어떤 형식으로 알려주면 되는지"를 항상 먼저 제시**해
  사용자가 형식을 몰라 막히지 않게 한다(빈 칸 채우기 방식, `docs/game_design.md`).
- **에셋·이름 식별**: 사용자는 원작 경험으로 (a) 지목된 에셋 파일의 인게임 용도, (b) 엔티티
  이름(드래곤/스킬/몬스터 — strings에 없음)을 매칭할 수 있다. ⇒ Claude는 막연히 묻지 말고
  **xlsx/마크다운 표 skeleton(파일경로·ID 열 + 사용자가 채울 purpose·name 열)을 만들어 제시**한다.

## 2. 절대 규칙 (HARD RULES)

1. **온라인 시스템은 만들지 않는다.** 네트워크/서버/로그인 코드 금지.
   PvP(콜로세움), 경매·거래, 친구, 서버 이벤트는 구현 대상이 아니다(스텁 처리).
2. **모든 상태는 로컬에 저장한다.** 진행도는 로컬 세이브 파일, 마스터 데이터는
   로컬 data 파일에서 읽는다.
3. **코드와 에셋을 분리한다.** 게임 로직(scripts/)과 원작 에셋(assets/raw/)을
   섞지 않는다. 나중에 에셋만 교체 가능해야 한다.
4. **작고 검증 가능한 단위로 작업한다.** "전투 시스템 전체"가 아니라
   "드래곤 1마리 idle 애니메이션 재생"처럼 실행해서 눈으로 확인 가능한 단위로.
5. **큰 변경 전에는 먼저 계획을 제시하고 승인을 받는다.** 바로 코드 작성 금지.
6. **수치/공식을 임의로 지어내지 않는다.** 전투 공식·확률·성장치는 추측 금지.
   출처(데이터 파일/위키/관찰)가 확인된 값만 쓰고, 미확인 값은 TODO로 표시한다.
7. **추측한 부분은 코드에 `# ASSUMPTION:` 주석으로 명시한다.**

## 4. 에셋 처리 메모

- **plist 아틀라스**: Cocos2d-x 포맷. Godot의 SpriteFrames/AtlasTexture로
  변환하는 도구를 `scripts/tools/`에 만들어 사용한다. 원본은 절대 수정하지 않는다.
- **Spine**: spine-godot 런타임으로 `assets/raw/spine/`의 json+atlas+png를 로드.
  드래곤별 애니메이션 키(idle/attack/hit 등)를 `dragons.json`에 매핑한다.
- **stringsData**: 텍스트 외에 게임 데이터가 섞여 있을 수 있으니 파싱 시
  데이터성 항목을 분리해 `data/`로 보낸다.

## 5. 데이터 모델 (서버 대체)

- **마스터 데이터**(불변): `data/*.json`에서 로드. 드래곤/스킬/스테이지/조합 규칙.
- **세이브 데이터**(가변): `saves/`에 저장. 보유 드래곤, 인벤토리, 재화,
  진행도, 부화 타이머 등. 저장/로드는 `scripts/core/save_system.gd` 단일 진입점으로.
- 가챠/부화 랜덤은 로컬 RNG로 재현. 시드 고정 옵션을 둬서 디버깅을 쉽게 한다.

## 10. 아키텍처 가이드라인 (계층 분리)

> 코드를 쓰기 전에 "이건 어느 계층인가"를 먼저 정한다. 아래 규칙은 §2 절대 규칙과
> 동급으로 취급한다. 위반이 불가피해 보이면 임의 진행 금지 — 먼저 사용자에게 확인.

### 10.1 3계층 분리 (data / logic / render)

이 프로젝트는 세 계층을 **명확히 분리**해서 구현한다.

- **data 층** — 정적 게임 정의. 드래곤 종족, 스킬 수치, 아이템 효과, 진화 트리,
  부화/교배 확률, 레벨업 곡선 등. **코드가 아니라 데이터 파일(`data/*.json`)**로 관리.
  위키·기억 기반으로 계속 보정되는 영역이므로 반드시 코드와 분리한다.
  (현재: `data/*.json` + `Data` 오토로드가 읽기 전용으로 로드)
- **logic 층** — 게임 규칙과 상태 변화. 부화·전투·교배·성장 등. **그래픽·화면·에셋을
  일절 몰라야** 하며, **화면 없이도 테스트 가능**해야 한다(헤드리스 단위 테스트).
  로직은 **"결과"만 산출**하고 연출은 하지 않는다.
  (위치: `scripts/systems/` + `scripts/core/`의 순수 규칙 코드)
- **render(scene) 층** — 에셋으로 화면을 그리고 입력을 받는 부분.
  (위치: `scenes/*.tscn` + `scripts/ui/`)

### 10.2 의존성 방향 (가장 중요 — 반드시 준수)

```
render  →  logic  →  data        (의존은 이 단방향으로만)
```

- 위 계층은 아래 계층을 알 수 있지만, **아래 계층은 위 계층을 절대 알면 안 된다.**
  즉 **data 층은 화면을 모르고, logic 층은 어떤 스프라이트가 그려지는지 모른다.**
- **역방향 의존 금지**: data나 logic이 render(노드/씬/스프라이트/`scenes`,`scripts/ui`)를
  import·참조·인스턴스화하는 것은 금지.
- logic은 Node 트리/AnimationPlayer/AudioStreamPlayer 등 표현용 엔진 객체에 의존하지 않는다.
  (순수 GDScript 클래스나 데이터 in/out 함수로 작성 → 헤드리스 검증 가능하게.)

### 10.3 명시적 금지 사항

- **로직 계층 안에 표현 로직을 끼워넣지 말 것.** 애니메이션 재생, 스프라이트 그리기,
  사운드 재생, 파티클·화면 연출은 logic에 두지 않는다. 데미지 계산·상태 변화 같은
  규칙은 **결과값만 반환**하고, 그 결과를 **render 층이 받아서 연출**한다.
  (예: 전투 로직은 `{damaged, crit, dead, ...}` 같은 결과/이벤트를 돌려주고,
  cave/battle 씬이 그걸 보고 hit 모션·데미지 숫자·사운드를 재생.)
- **편의를 위해 계층을 섞는 단축 구현 금지.** 계층 위반이 불가피해 보이면 **임의로
  진행하지 말고 먼저 사용자에게 확인**한다.

### 10.4 추가로 분리해 둘 보조 구조

- **에셋/리소스 관리**: "논리적 이름 ↔ 실제 파일" 매핑을 흡수하는 계층. logic은 파일
  경로를 모르고 논리적 키만 쓴다. **에셋 카탈로그를 이 매핑의 데이터로 사용**한다.
  (현재: `cocos_export.py`의 `_manifest.json`, `docs/asset_inventory.md`, 변환 산출물
  `assets/converted/`. render가 이 매핑을 통해 키→AtlasTexture를 로드.)
- **입력**: **입력을 받는 곳과 입력에 대한 처리를 분리**한다. (원시 입력 수집 →
  의미 있는 액션으로 변환 → logic 호출. 입력 코드에 게임 규칙을 직접 박지 않는다.)
- **상태/씬 관리**: 동굴/월드맵/전투/메뉴 등 **게임 모드를 명시적 상태 기계**로 관리.
  (전환 규칙을 한 곳에 모은 씬 매니저 — `scripts/core/`. 씬끼리 직접 서로를 로드·점프
  하지 않게.)
- **저장/영속성**: 유저 데이터 직렬화·역직렬화를 **한 곳에 모으고**, logic 층은 저장
  행위의 세부(파일 포맷/경로/시점)를 모르게 한다.
  (현재: `SaveSystem`=순수 파일IO, `UserDB`=유저 상태 스키마+API. 게임 시스템은
  UserDB API로만 상태를 바꾸고 직렬화 방식은 모른다.)
